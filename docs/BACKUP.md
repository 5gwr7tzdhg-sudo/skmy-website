# Резервное копирование SKMY

## PostgreSQL backup

Создавайте дамп базы в custom-формате PostgreSQL:

```bash
mkdir -p /srv/skmy/backups/postgres
pg_dump -h 127.0.0.1 -U skmy_user -Fc skmy_website_db > /srv/skmy/backups/postgres/skmy_$(date +%F_%H%M).dump
```

Запускайте команду от пользователя с доступом к БД. Не храните пароль в shell history; используйте защищённый `.pgpass` или безопасную переменную окружения.

## Восстановление PostgreSQL

Восстановление выполняйте только после подтверждения выбранного дампа и в окне обслуживания.

```bash
sudo -u postgres dropdb skmy_website_db
sudo -u postgres createdb --owner=skmy_user skmy_website_db
pg_restore -h 127.0.0.1 -U skmy_user -d skmy_website_db /srv/skmy/backups/postgres/skmy_YYYY-MM-DD_HHMM.dump
```

После восстановления проверьте вход в admin, список статей, новости и контактные обращения. Перед удалением или перезаписью рабочей базы всегда сделайте свежий backup.

## Backup uploads

Когда появится каталог пользовательских загрузок, например `/srv/skmy/skmy_website/uploads`, архивируйте его отдельно от дампа БД:

```bash
tar -czf /srv/skmy/backups/uploads/skmy_uploads_$(date +%F_%H%M).tar.gz /srv/skmy/skmy_website/uploads
```

Пока загрузка изображений не реализована, эту процедуру можно подготовить, но каталог может отсутствовать.

## Backup `.env`

`.env` содержит секреты. Храните зашифрованную резервную копию вне Git и вне публичного web root. Доступ к ней должен быть ограничен только ответственным администраторам.

Пример защищённого копирования на сервере:

```bash
install -m 600 /srv/skmy/skmy_website/.env /srv/skmy/backups/private/skmy_env_$(date +%F).backup
```

Для off-site backup используйте шифрование и отдельное защищённое хранилище.

## Рекомендуемая периодичность

- PostgreSQL: ежедневно, а также перед каждым обновлением или изменением данных.
- Uploads: ежедневно после появления пользовательских файлов.
- `.env`: после каждого изменения конфигурации и перед обновлением сервера.
- Проверка восстановления: не реже одного раза в квартал на отдельной тестовой базе.
- Хранение: минимум 7 ежедневных, 4 еженедельных и 3 ежемесячных backup-набора, если позволяют политика и объём хранилища.

Backup не считается рабочим, пока не была проверена процедура восстановления.
